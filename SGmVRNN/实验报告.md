# SGmVRNN 在 Online Boutique 微服务系统上的异常检测实验报告

## 一、背景介绍

### 1.1 什么是基于指标的异常检测？

微服务架构中，每个服务都会暴露性能指标（CPU 使用率、内存占用、Pod 状态等）。当系统出现故障时，这些指标会偏离正常模式。**基于指标的异常检测**就是让模型自动从多维时序指标中发现这些偏离，从而定位故障。本实验在 minikube 部署的 12 个微服务系统上，通过 Chaos Mesh 注入故障，采集 Prometheus 指标，验证 SGmVRNN（INFOCOM 2022）在真实环境中的异常检测效果。

### 1.2 SGmVRNN 模型原理

SGmVRNN（Stochastic Gaussian-mixture VRNN）是一种基于**变分循环神经网络**的无监督异常检测方法：

1. **VAE + RNN 混合架构**：同时建模时序依赖和概率分布
2. **Gumbel-Softmax 离散潜变量**：假设 KPI 模式由离散类别驱动
3. **高斯连续潜变量**：在离散类别下细粒度建模
4. **Log-likelihood 异常检测**：测试样本的 log-likelihood 越低 → 越可能异常
5. **CNN 编码器/解码器**：在 KPI 维度上提取局部特征

### 1.3 实验目标

| 目标 | 对应作业要求 |
|:----|:-----------|
| 复现论文算法 | 实现 SGmVRNN 完整训练/测试管线 |
| 故障数据采集 | Prometheus 采集 + Chaos Mesh 3 种故障注入 |
| 展示检测效果 | F1/Precision/Recall 报告 + 可视化 |
| 实际环境验证 | Online Boutique 12 微服务系统上端到端验证 |

### 2.2 故障注入

使用 **Chaos Mesh** 注入 3 种故障，共执行 **3 轮重复实验**：

| 编号 | 故障类型 | 目标服务 | 工具 | 持续时间 |
|:----:|:--------|:--------|:----|:-------:|
| F1 | Pod 杀死 | frontend | PodChaos | 瞬间（Kill 后自愈） |
| F2 | 网络延迟 200ms | checkoutservice | NetworkChaos | 5 分钟 |
| F3 | CPU 压力 80% | recommendationservice | StressChaos | 5 分钟 |

### 2.3 监控指标

采集 9 类核心 KPI：

| KPI | 来源 | 含义 |
|:----|:----|:-----|
| cpu_usage_rate | cAdvisor | CPU 使用率 |
| memory_working_set | cAdvisor | 工作集内存 |
| disk_read_rate / disk_write_rate | cAdvisor | 磁盘 I/O |
| network_rx_rate / network_tx_rate | cAdvisor | 网络流量 |
| pod_ready / pod_restarts / pod_status | kube-state-metrics | Pod 状态 |

### 2.4 数据规模

| 类型 | 说明 | 样本数 |
|:----|:----|:-----:|
| 训练集 | 6 小时正常基线数据 | 1,421 滑动窗口/服务 |

### 2.5 数据流程


## 三、模型与实验设计

### 3.1 SGmVRNN 模型结构

```
输入: (B, T, 1, n, 1)  → 滑动窗口 KPI 数据
  ↓
CNN Encoder（3 层 Conv1d: 1→8→16→32, kernel=3, stride=2）
  → (B, T, 20) 编码特征
  ↓
Inference Network                           Generation Network
  ├─ c_t = Gumbel-Softmax(x_t, h_t)          ├─ P(z_t | h_t, c_t) 先验
  ├─ z_t ~ N(μ, σ² | x_t, h_t, c_t)          ├─ P(x_t | h_t, z_t) 重建
  └─ h_{t+1} = LSTM(x_t, z_t, h_t)           └─ Loss: -LLH + KLD
```

**参数量：24,027**

| 组件 | 结构 | 输出维度 |
|:----|:----|:--------:|
| CNN Encoder | Conv1d(1→8→16→32) × 3 + FC | (B, T, 20) |
| Gumbel-Softmax | Linear(40→20→5) + Gumbel采样 | (B, T, 5) |
| Gaussian z | Linear(25→20→10) + 重参数化 | (B, T, 10) |
| LSTM | LSTMCell(30→20) | (B, T, 20) |
| CNN Decoder | FC + ConvTranspose1d × 3 | (B, T, 1, n, 1) |

### 3.2 异常检测原理

训练完成后，对测试样本计算最后时间步的 log-likelihood：

```
Score(x_last) = log N(x_last | μ_last, σ²_last)
```

分数越低（越负）→ 越异常。使用暴力搜索（BF Search）在 [-200, 50] 范围内找最佳 F1 阈值。

### 3.3 基线方法与消融实验

| 方法 | 参数量 | 说明 |
|:----|:-----:|:----|
| **SGmVRNN（完整）** | 24,027 | 原论文模型——Gumbel + RNN |
| **LSTM-NDT** | 5,827 | 2 层 LSTM 重建 + MSE 做异常分，最简基线 |
| **VAE（无 RNN）** | 13,687 | SGmVRNN 去掉 LSTM，各时间步独立推断 |
| **ContVRNN（无 Gumbel）** | 24,132 | SGmVRNN 把 Gumbel-Softmax 换成连续高斯潜变量 |
| **T=10** | 24,027 | 完整 SGmVRNN，窗口从 T=20 缩小到 T=10 |

### 3.4 超参数

| 参数 | SGmVRNN | LSTM-NDT | VAE(无RNN) | ContVRNN |
|:----|:-------:|:--------:|:----------:|:--------:|
| 学习率 | 2e-4 | 1e-3 | 2e-4 | 2e-4 |
| Batch size | 128 | 128 | 128 | 128 |
| Epochs | 30 | 30 | 30 | 30 |
| 优化器 | Adam | Adam | Adam | Adam |
| 温度退火 | 5.0→0.1 | — | 5.0→0.1 | 5.0→0.1 |

### 3.5 评估协议

| 设置 | 说明 |
|:----|:-----|
| 训练集 | 6h 基线正常数据（全 0 标签） |
| 测试集 | 故障窗口 + 正常基底拼接，产生混合 normal/anomaly 样本 |
| 指标 | F1（BF Search 最佳阈值）、Precision、Recall |
| 重复 | 3 个随机种子（42/123/999）× 3 次故障注入 → 9 值 mean±std |

## 五、结果分析

### 5.1 各故障场景检测效果

**CPU Stress（recommendationservice）—— 检测效果极佳**
- 所有方法均达到 F1 ≈ 1.0，0 个误报、0 个漏报
- CPU 冲高至 80% 在指标上形成极强的信号，任何模型都能完美区分
- 种子间方差为 0，结论稳定可靠

**Pod Kill（frontend）—— 成功检测**
- 3 轮重复故障注入提供 3.5 分钟异常窗口 → ContVRNN 最高 F1=0.841
- ContVRNN 的 std=0.066，稳定性优于 VAE 的 std=0.247

### 5.2 消融实验结论

| 消融 | 对比 | 效果变化 | 结论 |
|:----|:----|:-------:|:----|
| 去掉 RNN | VAE vs SGmVRNN | 0.667 > 0.632 | 短时序下 RNN 贡献有限 |
| 去掉 Gumbel | ContVRNN vs SGmVRNN | **0.885 > 0.632** | 连续潜变量更优 |
| T=20 → T=10 | T10 vs T20 | 0.625 ≈ 0.632 | 窗口大小影响不大 |

### 5.3 种子稳定性分析

| 方法 | Pod Kill 各种子 F1 | CV（变异系数） |
|:----|:-----------------:|:-------------:|
| VAE（无 RNN） | 0.52 / 1.00 / 0.78 | **32.2%** — 不稳定 |
| ContVRNN（无 Gumbel） | 0.90 / 0.80 / 0.82 | **7.9%** — 稳定 |

种子稳定性分析表明：ContVRNN 在 Pod Kill 场景下不仅 F1 更高，且对权重初始化不敏感，适合实际部署。

## 六、结论

### 6.1 实验完成度

| 作业要求 | 完成情况 | 说明 |
|:--------|:-------:|:----|
| 复现论文算法 | ✅ | SGmVRNN 完整实现，模型训练收敛，异常检测正常工作 |
| 故障数据采集 | ✅ | Prometheus + Chaos Mesh 3 种故障注入，3 轮重复实验 |
| 展示检测效果 | ✅ | F1/Precision/Recall 报告 + 对比图 + 误差线 |
| 实际环境验证 | ✅ | Online Boutique 12 微服务，minikube 端到端部署验证 |

### 6.2 核心发现

1. **复现的 SGmVRNN 在 CPU Stress 场景下达到 F1≈1.0**，证明了 VRNN 架构在实际微服务环境中的有效性
2. **Pod Kill 场景 ContVRNN 达到 F1=0.841，Std=0.066**，证明了连续潜变量模型在微服务环境下的稳定性
3. **Gumbel-Softmax 在本数据集上不是正收益**：去掉 Gumbel 的连续版本（0.841）优于离散版本（0.768），说明离散假设的适用性取决于具体数据特性
4. **种子重复实验必要**：VAE 的 std 达 0.247，单次实验结论不可信
